Platform Architecture

Monorepo Structure

The platform is an npm workspaces monorepo with a single shared package and five independent apps. All source code lives in one repository; each app deploys to a separate GitHub Pages site.

irc-emergency-platform/ ├── packages/ │ └── shared/ @irc/shared — data, utilities, UI, Supabase client │ └── apps/ ├── classification/ @irc/classification — vanilla JS ├── crf-calculator/ @irc/crf-calculator — vanilla JS ├── navigator/ @irc/navigator — React + TypeScript + Vite ├── admin/ @irc/admin — vanilla JS └── landing/ @irc/landing — static HTML

All app packages declare "@irc/shared": "*" as a dependency, resolved by npm workspaces to the local packages/shared/ directory.

The IIFE Bundle Pattern

The shared package is distributed to vanilla JS apps as an Immediately Invoked Function Expression (IIFE) bundle, available globally as window.IRCShared.

  1. packages/shared/build.js uses esbuild to bundle src/index.js as an IIFE with globalName: 'IRCShared'
  2. Each vanilla app's build.js copies the built bundle into its own directory
  3. HTML files load it via <script src="irc-shared.iife.js">, attaching everything onto window.IRCShared
  4. A shim script (irc-platform.js) then maps selected properties onto window.IRC for app-level use

The Navigator (React/Vite) imports @irc/shared as ES modules directly — it does not use the IIFE bundle. Vite resolves the workspace dependency at build time.

Two namespaces coexist in vanilla apps: window.IRCShared (raw IIFE output) and window.IRC (app-level namespace assembled by the shim + supabase.js + shared.js).

STANCE_MATRIX

A 3-dimensional lookup table: STANCE_MATRIX[severity][pv][rob] → 'white'|'yellow'|'orange'|'red'.

  • Severity: 1–10 (derived from affected population data)
  • PV (Population Vulnerability): 1–4 (country-level)
  • ROB (Response Operational Burden): 0–5 (country-level)

Severities 1–7 generally produce yellow; 6+ with higher PV/ROB escalate to orange; 8–10 produce red or orange depending on the country profile.

COUNTRIES Data

Array of 195 entries: [name, population, pv, rob]. Examples:

["Afghanistan", 42647492, 4, 5]
["South Sudan", 11088796, 4, 5]
["United Kingdom", 67736802, 1, 0]

Frontend Stack

Emergency Classification Vanilla JS

Multi-page app with no framework. Entry points: index.html (dashboard), classify.html (form), maps.html (Leaflet maps).

  • CSS: styles.css + shared design-tokens.css, flat CSS with custom properties
  • JS patterns: IIFEs, closures for state, all globals via window.IRC.*
  • Map: Leaflet.js v1.9.4 (CDN) with country markers
  • Key UI: Dashboard with time/stance/status filter pills, searchable country dropdown, severity scoring display

CRF Allocation Calculator Vanilla JS

Single-page calculator in one HTML file (allocation-calculator.html). ~2,500 lines including ~800 lines of inline CSS.

  • Calculation engine: Severity scoring (dual-pass: absolute + proportional), stance matrix lookup, per-client cost model ($47.34 × 6 months)
  • Coverage tiers: 1%, 2%, 3% of affected population with CRF/Gap split visualization
  • Proportional discount: Toggle that scales CRF allocation when target population is reduced below the 10% baseline
  • Scenario comparison: Save up to 3 scenarios, tabular comparison with max-highlighting
  • Auto-population: Fetches classifications from Supabase on load, auto-fills affected population and stance from active classifications

Response Navigator React

Single-page application built with React 19, TypeScript, Vite 6, and Tailwind CSS 3.

  • Routing: React Router DOM 7 with basename: '/emergency-response-navigator'
  • Pages: Home (role selector, country search, phase/sector grid), Navigator (filterable task browser), Resources (resource library)
  • State: ClassificationProvider context for country/stance/classification data, persisted to localStorage
  • Data source: process-data.json — large JSON file built from CSV sources at build time, bundled as a separate Vite chunk
  • SPA on GitHub Pages: Uses the standard 404.html redirect hack with sessionStorage.redirect
  • Service Worker: Registered at /emergency-response-navigator/sw.js
  • Manual chunks: process-data (data) and react-vendor (React + Router)

Platform Admin Vanilla JS

Single-page admin panel. Hash-based tab navigation (#data). Provides CSV/Excel import with drag-and-drop, column mapping UI, and preview. Shares the Supabase data-access layer from the Classification app.

Backend & Database

Supabase (PostgreSQL) Supabase

The platform uses Supabase as its backend — a hosted PostgreSQL database with a PostgREST API layer. No Supabase JS SDK is used; all calls are direct fetch() against the REST endpoint.

Security note: Row Level Security is enabled but with a fully permissive policy (FOR ALL USING (true)). This is by design for an internal proof-of-concept tool.

Data Access Patterns

  • Read (shared): fetchClassifications() paginates with limit=1000&offset=N until a page returns fewer than 1000 rows. Maps snake_case → camelCase via mapToCamel().
  • Write (classification app): IRC.db.save(record) maps camelCase → snake_case via mapToSnake() (with type coercion, date normalization, empty-string → null), then POSTs with Prefer: return=representation,resolution=merge-duplicates.
  • Bulk import (admin app): IRC.db.importBatch() inserts in chunks of 200, diffs against existing records, deletes removed entries in batches of 100.
  • Caching: Write-through cache to localStorage['irc_classifications'] for offline fallback.

Field Mapping

27 camelCase ↔ snake_case field pairs defined in TO_SNAKE / TO_CAMEL constants. The mapToSnake() function also performs:

  • Empty-string → null coercion for constrained columns
  • String → boolean for ipc4_used
  • parseFloat for total_affected
  • Date validation/normalization to YYYY-MM-DD
  • JSON.parse for JSONB fields

AI Integrations

Ask Albert AI

Albert is the platform's AI assistant, available in all four apps. It provides context-aware help about emergency classifications, CRF funding, and response protocols.

Architecture

All Albert implementations connect to a single Supabase Edge Function:

POST https://qykjjfbdvwqxqmsgiebs.supabase.co/functions/v1/chat

No API key is required on the client. The Edge Function holds the OpenRouter key server-side and handles RAG (Retrieval-Augmented Generation), system prompt construction, and classification data fetching.

Request Payload

{
  "messages": [{ "role": "user", "content": "..." }],
  "site": "classification" | "crf" | "navigator",
  "model": "google/gemini-2.5-flash"
}

The site field tells the Edge Function which tool is being used, allowing it to tailor the system prompt and RAG context accordingly.

Streaming Protocol

Responses are delivered as Server-Sent Events (SSE). The client reads the response body as a stream, splits on newlines, processes data: {...} lines, extracts choices[0].delta.content, and accumulates the streamed text. The stream terminates with data: [DONE].

Available Models

Model IDNameRole
google/gemini-2.5-flashGemini 2.5 FlashDefault — fast, low cost
anthropic/claude-sonnet-4Claude Sonnet 4Better reasoning for complex questions
openai/gpt-4.1-miniGPT-4.1 MiniBalanced speed and quality

Model selection is persisted to localStorage['irc_openrouter_model'] and sent with each request.

Dynamic Example Queries

Both the vanilla and React Albert implementations generate context-aware example queries from live classification data:

  1. Filter classifications to non-expired, red or orange stance
  2. Sort by severity (desc), then date (desc)
  3. Take up to 2 unique countries
  4. Generate queries like "Tell me about the Sudan Red classification"
  5. Pad with static fallback queries specific to each app

Feedback System (Navigator)

The Navigator's chat panel includes a thumbs-up/thumbs-down feedback mechanism. Feedback is sent as a fire-and-forget POST to the same Edge Function endpoint with a { feedback: { messageId, rating, query, timestamp } } payload.

  3. User selects country in CountrySearch → filters to active classifications for that country
  4. highestStakes() auto-selects the most severe classification
  5. Classification context (country + stance) is available to all components via useClassification() hook

Cross-App Navigation

site-config.js is the universal navigation contract. Every app calls getSiteConfig() to get environment-correct URLs for all other apps. The shared header's site-switcher dropdown is built from this config. A single change to site-config.js propagates URL changes everywhere.

CI/CD Pipeline

Defined in .github/workflows/deploy.yml. Triggered on push to main or manual dispatch.

Build Sequence

# 1. Install
npm ci

# 2. Build shared package (esbuild IIFE)
npm run build:shared

# 3. Build each app (copies IIFE bundle + assets)
npm run build:classification
npm run build:navigator      # Vite build → dist/
npm run build:crf
npm run build:admin
npm run build:landing

# 4. Deploy each app to its standalone repo's gh-pages branch
peaceiris/actions-gh-pages@v3  # × 5 apps

Each deploy uses a DEPLOY_TOKEN personal access token secret to push to the external repositories. The IIFE bundle is gitignored — CI builds it fresh every time.

Navigator Build Pipeline

The Navigator has an additional pre-build step: build-data.mjs parses CSV source files from a csv/ directory into process-data.json. The CSVs are not committed; in CI, the script detects no data and preserves the existing committed JSON. This means task data can only be updated by running the build locally with CSV files present.

Key Design Decisions

  • No framework for Classification & CRF: Vanilla JS keeps the apps dependency-free, fast-loading, and simple to deploy as static files. The tradeoff is more manual DOM manipulation.
  • React for Navigator only: The Navigator's task filtering, role-based views, and context management benefit from React's component model and state management.
  • Direct PostgREST instead of Supabase SDK: Raw fetch() avoids a ~40KB SDK dependency and keeps the vanilla apps truly dependency-free.
  • Server-side AI key: The Supabase Edge Function holds the OpenRouter API key, so no AI credentials are exposed in client-side code.
  • IIFE for code sharing: The IIFE pattern allows vanilla apps to consume the shared package via a simple <script> tag, while the Navigator uses the same source via ES module imports.

Known Patterns

  • Duplicated field mappings: TO_SNAKE/TO_CAMEL exist in both packages/shared/src/supabase.js and apps/classification/supabase.js. The app version extends them with the full IRC.db.* API.
  • window.__askAlbert bridge: The Navigator injects a global function to let child components open the chat panel without prop-drilling.
  • SPA redirect hack: The Navigator uses a 404.html page that stores the path in sessionStorage and redirects to /, where main.tsx reads it and replaces the history state. Standard pattern for SPAs on GitHub Pages.
